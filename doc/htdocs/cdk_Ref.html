Introduction

The project reference is supposed to be a compendium for all kind of information related to the CDK project, as there are questions regarding the program design, how to document things or what to do in order to update the web pages.

Design Considerations

The most important aspect in the design of the CDK is the clean separation of function. We decided to use the meanwhile classical model-view-controler design pattern. (see for example this article on CompSci and the references therein for more). The basic concept is that there is a single MODEL that contains the data. In our case the data would be an internal representation of a molecule, a chemical reaction, an ensemble of molcecules, etc.

There can be multiple VIEWs of the same model

  • a 2D representation like the one in JChemPaint, consisting mainly of lines, representing bonds and some Elementsymbols, or

  • a 3D representation, rotatable, nicely shaded with perspective scaling.

The CONTROLER gives the user means of manipulating how the VIEW of the MODEL is supposed to look like, e.g. for rotating a molecule in the 3D view. But is also may provide a way of manipulating the model, so that a user of JChemPaint can, for example, remove a bond from a molecule, etc.

Cleanly separating these parts, using well-defined interfaces between them, gives you great flexibility. Making an applet out of JChemPaint, i.e. plugin a Java 1.1.8 GUI where there is now a Browser-incompatible Java 2 interface, would be much easier if JCP was nicely designed using the MVC paradigm.

The MODEL in the CDK consists of

  • Java classes for chemical/physical concepts, like:

    • Element

    • Isotope

    • Atom

    • Bond

    • Molecule

    • Ring

    • etc.

  • Java classes that allow for the administration of the above:

    • ChemModel (may contain all of the above)

    • ChemSequence (contains a sequence of ChemModels, e.g. to represent the course of a reaction, where reactants, intermediates and products are stored in separated ChemModels)

    • ChemFile (a collection of ChemModels and/or ChemSequences)

  • Java finder classes for the production of value-added representations from concepts in a ChemModel

    • RingFinder, for example, would produce a SetOfRings, when given a molecule.

    • BendFinder would produce a list of Bends (a Bend is an ensemble of three atoms from a molecule that make up an angle)

The full concept can be found on the photograph that I send you in the first mail and in Egons UML diagrams. The VIEW and the CONTROLER are now in the responsibility of the particular programming team, in our case the JChemPaint team. The JChemPaint team, for example, would produce a 2DRenderer.class which creates the regular 2D drawing program look from the underlying model. The JCP controller would add mouse control for drawing and deleting bonds, rotating molecules, etc.
